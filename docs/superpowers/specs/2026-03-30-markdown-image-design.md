# Markdown 长图页面设计

## 背景

当前项目是一个在线剪贴板共享应用，前端基于 Vue 2 和 Vuetify。现有导航内包含剪贴板和图片识别等页面，但缺少一个适合临时编辑 Markdown 并直接导出图片的能力。用户要求新增独立菜单和页面，支持在线编辑 Markdown，支持整篇内容导出为图片，导出尺寸应适配当前设备，或允许用户手动选择移动端或 PC 版式。同时，若相关第三方库体积较大，则必须在用户访问该菜单后再按需加载，首页默认不加载。

## 目标

新增一个独立的 Markdown 长图页面，满足以下目标：

1. 用户可在浏览器中直接编辑 Markdown 文本。
2. 页面可实时预览 Markdown 渲染结果。
3. 用户可将整篇预览内容导出为一张 PNG 长图。
4. 导出时支持三种布局模式：当前设备、移动端、PC 端。
5. Markdown 解析和图片生成相关依赖不进入首页首屏包，只有在进入该页面后才加载。

## 非目标

以下内容不纳入本次范围：

1. 不新增后端接口，也不做服务端渲染。
2. 不做多人协作、草稿云同步、版本历史等编辑器能力。
3. 不做 PDF 导出。
4. 不引入复杂的 Markdown 扩展语法体系，例如 Mermaid、数学公式、流程图等。

## 方案选择

### 推荐方案

使用 `markdown-it` 负责 Markdown 到 HTML 的解析，使用 `html-to-image` 负责将预览区域导出为 PNG 图片。

选择原因如下：

1. `markdown-it` 解析性能好、浏览器兼容性成熟，足以覆盖标题、段落、列表、引用、分隔线、代码块、链接、图片等常用 Markdown 功能。
2. `html-to-image` 直接面向 DOM 导出图片，适合将预览区域以实际样式生成长图，清晰度和输出控制能力都足够。
3. 两者都可以通过动态 `import()` 按需加载，满足首页不加载相关依赖的要求。
4. 相比 `remark/rehype` 体系，当前功能不需要 AST 级复杂扩展，较轻的实现更符合现有项目体量。

### 放弃方案

1. 使用 `remark/rehype`：可扩展性更强，但依赖链更重，接入复杂度更高，对当前需求属于过度设计。
2. 通过服务端或无头浏览器生成图片：一致性高，但需要额外服务端能力与部署支持，不符合本项目现状。

## 前端架构变更

### 导航与路由

在左侧导航菜单中新增一个菜单项，例如“Markdown 长图”。

路由层新增一个独立页面：

1. 新页面位于 `client/src/views/Markdown.vue`。
2. 路由使用动态 `import()` 实现页面级懒加载。
3. 为避免把新页面打入首页 chunk，现有静态导入的页面路由可一并调整为懒加载，或至少保证 Markdown 页面独立拆包。

### 页面布局

Markdown 页面由三个区域组成：

1. 编辑区：多行文本输入框，允许直接编写 Markdown。
2. 控制区：提供导出模式选择、导出按钮、重置示例内容等操作。
3. 预览区：将 Markdown 渲染为文章样式内容，作为图片导出的源节点。

布局策略：

1. 在桌面端采用左右双栏布局，左侧编辑，右侧预览。
2. 在移动端采用上下布局，先编辑后预览。
3. 预览区宽度由当前导出模式驱动，保证用户看到的版式与导出结果一致。

## 数据流与状态设计

页面内部维护以下状态：

1. `markdownText`：当前编辑中的 Markdown 文本。
2. `renderedHtml`：Markdown 解析后的 HTML 字符串。
3. `previewMode`：导出模式，取值为 `current`、`mobile`、`desktop`。
4. `loadingParser`：Markdown 解析库是否正在加载。
5. `exportingImage`：图片导出是否进行中。

数据流如下：

1. 用户进入页面后，页面按需加载 Markdown 解析库。
2. 页面用默认示例内容初始化 `markdownText`。
3. 用户输入文本后，通过轻量防抖触发解析，生成 `renderedHtml`。
4. 预览区根据 `renderedHtml` 与当前模式宽度实时展示最终版式。
5. 用户点击导出按钮后，再按需加载图片导出库，将完整预览节点导出为 PNG。

## 导出尺寸与清晰度策略

### 导出模式

提供三种模式：

1. `当前设备`：基于当前页面可用宽度计算预览宽度，并结合浏览器 `devicePixelRatio` 输出图片。该模式用于快速得到与当前设备视觉一致的结果。
2. `移动端`：使用固定移动端内容宽度，例如 390px 左右的设计宽度，适合生成手机阅读长图。
3. `PC 端`：使用固定桌面端内容宽度，例如 960px 左右的设计宽度，适合生成桌面版式长图。

### 输出清晰度

导出时图片缩放规则如下：

1. 默认使用 `window.devicePixelRatio` 作为基础像素比。
2. 为兼顾性能，对像素比设置上限，避免超高 DPR 设备在长内容场景下产生过大的内存压力。
3. 生成图片时使用白色背景，避免透明背景影响跨平台查看。
4. 导出文件名包含日期和模式标识，方便区分。

### 长图完整性

导出对象必须是整篇预览内容，而不是当前视口截图。实现上应确保：

1. 获取预览节点完整高度。
2. 在导出前临时固定导出宽度，确保节点按目标版式重新排版。
3. 生成图片后恢复页面显示状态，避免用户界面错乱。

## 样式与兼容性要求

Markdown 预览样式遵循“文章”展示方式，强调可读性和稳定性：

1. 标题、段落、列表、引用、分割线、代码块、行内代码、链接需要有明确样式。
2. 代码块应支持横向滚动，避免长代码挤坏布局。
3. 图片元素在预览中最大宽度不超过容器宽度。
4. 样式应避免依赖高级实验性 CSS 特性，以提升兼容性。
5. 页面在浅色和深色主题下都应可用，但导出的图片统一采用浅色文章底，以保持跨平台一致性。

## 性能策略

为满足“首页默认不加载”的要求，本次实现采用多层按需加载：

1. Markdown 页面本身通过路由懒加载。
2. `markdown-it` 在页面首次进入时动态导入。
3. `html-to-image` 在首次点击导出按钮时动态导入，而不是进入页面立即加载。

同时补充以下性能措施：

1. Markdown 解析增加短时间防抖，减少连续输入时的重复解析。
2. 仅在源文本变化后重新渲染 HTML，避免无意义计算。
3. 导出期间禁用重复点击，显示加载状态。
4. 对输出像素比设置合理上限，降低极长内容场景的浏览器卡顿风险。

## 错误处理

需要处理以下异常场景：

1. 解析库或导出库加载失败：提示用户稍后重试。
2. Markdown 解析异常：保底渲染为空内容或转为纯文本错误提示，不让页面崩溃。
3. 导出失败：提示失败信息，并恢复按钮状态。
4. 导出节点不存在或尚未完成渲染：阻止导出并给出提示。

## 测试与验证策略

实现完成后至少验证以下内容：

1. 首页加载时不会请求 Markdown 页面对应的异步 chunk。
2. 点击菜单进入新页面后，相关 chunk 才开始加载。
3. 标题、列表、引用、代码块、链接等常见 Markdown 能正确预览。
4. `当前设备`、`移动端`、`PC 端` 三种模式均可导出 PNG。
5. 导出结果为整篇长图，不截断底部内容。
6. 在移动端宽度下页面布局可正常使用。

## 实施范围

预计改动文件包括：

1. `client/package.json`：新增 Markdown 与图片导出依赖。
2. `client/src/router/index.js`：新增 Markdown 页面路由并调整懒加载方式。
3. `client/src/App.vue`：新增导航菜单入口与图标。
4. `client/src/views/Markdown.vue`：实现编辑、预览、导出主页面。
5. 如有必要，可新增轻量工具函数文件用于动态加载或导出参数封装，但应保持职责清晰。

## 风险与约束

1. 超长 Markdown 在高 DPR 设备上导出时可能带来明显内存压力，因此必须限制缩放倍率。
2. DOM 转图片类库对某些复杂外链字体或跨域图片敏感，本次优先保证纯文本与常规文章样式稳定导出。
3. 若 Markdown 中包含跨域远程图片，浏览器可能因 canvas 污染限制导致导出失败。本次默认不额外代理远程图片，页面提示以本地或同源资源为佳。

## 结论

本次功能以单独页面形式接入现有 Vue 2 项目，通过页面级和库级双重懒加载控制体积开销，使用 `markdown-it` 和 `html-to-image` 组合实现整篇 Markdown 长图编辑与导出。方案能够在较低实现成本下满足性能、兼容性和使用体验要求，并且不需要变更后端。
