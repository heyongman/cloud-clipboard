# 代理订阅转换页面设计

## 背景

当前项目是一个在线剪贴板共享应用，前端基于 Vue 2 和 Vuetify，后端基于 Node.js 和 Koa。项目现有菜单以文本、文件、OCR、Markdown 等轻量工具页为主，但尚未提供代理订阅处理能力。

本次需求是在现有项目中新增一个独立菜单和页面，用于维护一份全局代理订阅转换配置。用户可以录入一个或多个上游订阅 URL，通过后端拉取订阅内容，按正则规则过滤和屏蔽节点，并对外暴露一个固定不变的订阅地址。该地址不依赖现有管理员密码，而使用独立 token 鉴权；除非主动重置 token，否则订阅地址保持不变。配置更新后，后续访问该订阅地址应立即生效。

## 目标

本次设计需要满足以下目标：

1. 在前端新增一个独立菜单“订阅转换”和对应页面。
2. 页面维护唯一一份全局订阅转换配置，而不是创建多个任务或多个配置档。
3. 支持填写多个上游订阅 URL。
4. 后端支持尽量兼容多种输入来源，但统一输出为 Clash YAML。
5. 支持通过正则对节点名称执行“包含过滤”和“屏蔽过滤”。
6. 将过滤后的节点整理为一个单独代理组，并在最终配置中默认使用该组。
7. 删除上游订阅中其他多余代理组，仅保留“订阅节点”“自动选择”“故障转移”。
8. 生成一个固定订阅地址，前端支持一键复制。
9. 固定订阅地址使用独立 token 鉴权，并支持在管理页重置 token。
10. 用户修改上游源或规则后，无需更换订阅地址，下次访问即可获取最新结果。

## 非目标

以下内容不在本次范围内：

1. 不实现多个订阅配置档，也不提供订阅配置列表页。
2. 不实现规则订阅、规则集托管、脚本注入、策略脚本等高级 Clash 生态能力。
3. 不保证兼容所有订阅格式；重点覆盖 Clash YAML、Base64 节点订阅、常见 URI 列表。
4. 不引入外部订阅转换服务或重量级转换内核。
5. 不在本次实现中增加复杂的前端单元测试体系。

## 方案选择

### 推荐方案

采用“混合解析，统一输出 Clash YAML”的方案。

核心思路如下：

1. 后端拉取多个上游订阅源。
2. 自动识别输入内容是 Clash YAML、Base64 节点订阅，还是明文 URI 列表。
3. 将所有来源统一解析为内部代理节点数组。
4. 对节点去重并应用正则过滤规则。
5. 重新组装为一份精简的 Clash YAML，并通过固定订阅地址输出。

选择该方案的原因如下：

1. 能满足“输入尽量兼容多种来源，输出统一 Clash”的目标。
2. 与当前项目轻量单体结构更匹配，无需额外服务。
3. 可以把复杂性控制在“输入归一化”和“最终 YAML 生成”两块核心逻辑中，便于维护和测试。

### 放弃方案

1. 只支持 Clash YAML 输入：实现简单，但实用性不足，无法满足“支持其他格式更好”的要求。
2. 接入完整外部转换内核或第三方服务：格式覆盖可能更高，但依赖重、部署复杂，不符合当前项目体量。

## 总体架构

### 前端

前端新增一个独立页面用于管理唯一一份全局订阅转换配置。页面职责包括：

1. 展示固定订阅地址。
2. 管理上游订阅 URL 列表。
3. 管理包含正则和屏蔽正则。
4. 触发预览解析。
5. 保存配置。
6. 重置访问 token。
7. 一键复制固定订阅地址。

### 后端

后端新增一组与订阅转换相关的管理接口，以及一个面向 Clash 客户端的公开订阅接口。

后端职责包括：

1. 持久化保存全局订阅配置。
2. 校验和编译正则规则。
3. 拉取上游订阅内容。
4. 识别并解析多种订阅格式。
5. 合并、去重、过滤节点。
6. 生成精简的 Clash YAML。
7. 以独立 token 保护公开订阅接口。
8. 提供短时缓存，降低上游压力。

## 页面与交互设计

### 导航与路由

在左侧导航菜单中新增一个菜单项，例如“订阅转换”，并新增对应路由页面，例如 `client/src/views/Subscription.vue`。

路由采用动态导入，与当前其他工具页保持一致。

### 页面结构

页面采用单页表单布局，由四个主要区域组成：

1. 固定订阅地址卡片。
2. 上游订阅 URL 配置区。
3. 节点过滤规则配置区。
4. 预览结果与保存操作区。

### 固定订阅地址卡片

卡片展示以下信息和操作：

1. 当前固定订阅地址。
2. 一键复制按钮。
3. 重置访问密钥按钮。
4. 可选的说明文案，例如“修改规则或订阅源后，此地址保持不变，后续访问即时生效”。

### 上游订阅 URL 配置区

使用多行输入框，每行表示一个 URL。

行为约定如下：

1. 空行自动忽略。
2. 保存时做基本 URL 合法性校验。
3. 不要求保存时逐条探测可达性，避免因为上游暂时超时而阻塞配置更新。

### 节点过滤规则配置区

提供两个多行输入框：

1. 包含正则。
2. 屏蔽正则。

行为约定如下：

1. 每行一条规则。
2. 规则默认作用于节点名称，不直接匹配 server、port 或其他字段。
3. 包含规则为空时，表示默认放行全部节点。
4. 节点只要命中任一屏蔽规则，就必须剔除。
5. 节点在“包含规则”判断通过后，仍需继续执行“屏蔽规则”判断。

### 预览与保存区

提供以下动作：

1. 预览按钮：请求后端拉取并解析上游订阅，返回节点统计和结果样本。
2. 保存按钮：保存当前配置并清理订阅缓存。

预览结果建议展示：

1. 成功拉取的上游数量。
2. 拉取失败的上游数量。
3. 原始节点总数。
4. 去重后节点数。
5. 过滤后保留节点数。
6. 最终保留节点名称列表。
7. 错误摘要。

## 配置模型与持久化

### 配置模型

系统仅保存一份全局订阅转换配置，建议包含以下字段：

1. `sources`: 字符串数组，保存上游订阅 URL。
2. `includePatterns`: 字符串数组，保存包含正则。
3. `excludePatterns`: 字符串数组，保存屏蔽正则。
4. `token`: 字符串，保存公开订阅接口使用的独立访问密钥。
5. `updatedAt`: 数字时间戳，记录最近一次配置更新时间。

如需补充管理态信息，可增加非核心字段，例如最近一次预览时间或最近一次生成错误，但这些字段不应影响订阅输出协议。

### 持久化方式

建议在 `server-node` 侧新增独立 JSON 文件保存该配置，风格与现有 `history.json` 类似，不混入消息历史文件。

设计要求如下：

1. 服务启动时自动读取配置文件。
2. 若配置文件不存在，则自动生成默认配置。
3. token 默认在首次创建配置时生成。
4. 修改配置时整体覆盖写回，保持实现简单清晰。

## 管理接口设计

管理接口沿用现有管理员鉴权，即复用当前 `Authorization: Bearer <authCode>` 机制。

建议新增以下接口：

### `GET /subscription/config`

返回当前全局订阅配置，用于页面初始化。

返回内容应包含：

1. 上游订阅 URL 列表。
2. 包含正则列表。
3. 屏蔽正则列表。
4. 当前 token。
5. 固定订阅地址。
6. 最近更新时间。

### `PUT /subscription/config`

保存当前全局订阅配置。

行为要求如下：

1. 校验 `sources` 为非空数组。
2. 去除首尾空白和空行。
3. 对每条 URL 做基本格式校验。
4. 对每条正则做编译校验。
5. 保存成功后立即清空订阅缓存。

### `POST /subscription/preview`

按当前页面提交的配置即时执行一次预览，不要求先保存。

行为要求如下：

1. 使用请求体中的 URL 和正则作为临时配置。
2. 拉取和解析上游订阅。
3. 返回统计信息、节点列表和错误摘要。
4. 不写入持久化配置。

### `POST /subscription/token/reset`

生成新的 token，并立即持久化。

行为要求如下：

1. 返回新的固定订阅地址。
2. 重置后旧 token 立即失效。
3. 不修改 URL、正则等其他配置。

## 公开订阅接口设计

建议新增公开接口：

### `GET /subscription/clash?token=<token>`

该接口用于 Clash 客户端订阅。

行为要求如下：

1. token 缺失或错误时返回 `403`。
2. token 正确时，根据当前持久化配置实时生成结果。
3. 接口路径固定，只有 token 变化时最终订阅地址才变化。
4. 返回内容类型应为文本，适合 Clash 直接导入。

## 数据流设计

公开订阅接口的数据处理顺序固定如下：

1. 读取当前持久化配置。
2. 校验 token。
3. 按顺序拉取所有上游订阅 URL。
4. 对每个响应内容做格式识别和解析。
5. 将解析结果合并为统一节点数组。
6. 执行去重。
7. 执行包含规则过滤。
8. 执行屏蔽规则过滤。
9. 组装精简 Clash YAML。
10. 返回结果文本。

管理页预览接口走相同转换流程，但使用请求体中的临时配置，且不依赖持久化 token。

## 输入格式兼容策略

首批实现重点支持以下输入类型：

1. Clash YAML。
2. Base64 编码的节点订阅。
3. 明文 URI 列表。

### Clash YAML

识别策略：

1. 内容可被 YAML 解析。
2. 包含 `proxies` 数组。

处理方式：

1. 提取 `proxies` 数组作为节点来源。
2. 忽略上游的 `proxy-groups`、`rules`、`rule-providers` 等非节点信息。

### Base64 节点订阅

识别策略：

1. 原始响应不符合 Clash YAML。
2. 内容可被 Base64 解码为文本。
3. 解码后文本中包含多行代理 URI。

处理方式：

1. 逐行切分。
2. 解析常见代理链接，转换为统一内部结构。

### 明文 URI 列表

识别策略：

1. 内容本身是多行文本。
2. 每行以常见代理协议前缀开头。

处理方式：

1. 跳过空行。
2. 逐条解析为内部代理节点结构。

## 节点归一化与去重

### 内部节点结构

无论输入来源如何，解析完成后都应归一化为统一节点对象，再进入后续流程。

内部对象至少应包含：

1. 节点名称。
2. 协议类型。
3. 服务器地址。
4. 端口。
5. 协议所需的其他参数。

### 去重策略

建议使用“协议 + 服务器 + 端口 + 节点名称”作为去重键。

原因如下：

1. 同一订阅源或多个订阅源可能包含重复节点。
2. 仅按名称去重可能误伤不同服务器。
3. 仅按地址去重可能误伤同地址不同配置节点。

## 过滤规则设计

### 包含规则

1. 包含规则按行拆分。
2. 任意一条规则命中节点名称，即视为通过包含过滤。
3. 如果包含规则为空，则所有节点默认通过包含过滤。

### 屏蔽规则

1. 屏蔽规则按行拆分。
2. 任意一条规则命中节点名称，即视为被剔除。
3. 屏蔽规则优先级高于包含规则；即便节点已被包含，也可被屏蔽规则删除。

### 非法正则

1. 保存配置和预览配置时都必须先尝试编译所有正则。
2. 若存在非法正则，接口应返回明确错误，并指出具体是哪一条规则无效。

## Clash 输出结构设计

### 节点输出

最终输出仅保留过滤后的节点，不保留上游其他附加结构。

### 代理组输出

最终配置只保留三个代理组：

1. `订阅节点`
2. `自动选择`
3. `故障转移`

其中：

1. `订阅节点` 包含全部过滤后的节点。
2. `自动选择` 和 `故障转移` 的候选节点都来自 `订阅节点` 这批节点。
3. 默认策略应优先指向 `订阅节点`，保证客户端导入后直接使用处理后的分组。

### 其他 Clash 字段

为确保输出结果简洁稳定，可仅生成订阅所需的基础字段，不继承上游无关配置。具体字段集合应保持最小可用原则，不在本次设计中扩展复杂规则系统。

## 缓存策略

### 目标

因为 Clash 客户端可能在短时间内重复访问订阅地址，接口需要设置一个很短的内存缓存以降低对上游的请求压力。

### 策略

1. 公开订阅接口按当前配置结果做短时缓存，例如 30 秒。
2. 管理页执行保存配置后，立即清空缓存。
3. 管理页执行预览时，不复用公开订阅缓存，避免预览结果与正在使用的正式配置混淆。

### 约束

缓存只存在于进程内存中，不做磁盘级缓存，也不做历史版本回滚。

## 错误处理

### 管理接口

1. URL 非法时返回 `400`。
2. 正则非法时返回 `400`，并指出具体规则。
3. 保存接口只校验格式，不强制即时拉取全部上游内容。

### 预览接口

1. 如果部分上游拉取失败，但仍有可用节点，则返回成功结果，并附带错误摘要。
2. 如果全部上游都失败，则返回明确错误。
3. 如果解析成功但过滤后无节点，则返回明确错误，提示过滤结果为空。

### 公开订阅接口

1. token 错误返回 `403`。
2. 全部上游失败时返回文本错误，不返回空 YAML。
3. 过滤后无节点时返回文本错误，不返回空 YAML。
4. 如果只有部分上游失败但仍能生成节点，则继续输出订阅结果。

## 安全要求

1. 管理接口继续使用现有管理员鉴权。
2. 公开订阅接口必须与管理员密码隔离，使用独立 token。
3. 页面上展示 token 时不需要额外掩码，因为其主要目的是便于复制订阅地址，但应避免在无鉴权接口中泄露。
4. token 重置后旧 token 立即失效。

## 测试与验证策略

### 后端逻辑测试

优先为纯逻辑部分增加自动化测试，覆盖以下内容：

1. 解析 Clash YAML。
2. 解析 Base64 节点订阅。
3. 解析明文 URI 列表。
4. 节点归一化与去重。
5. 包含规则和屏蔽规则。
6. 过滤后生成精简 Clash YAML。
7. token 校验和错误分支。

### 前端交互验证

本次以前端交互自测为主，至少验证以下内容：

1. 页面首次加载能正确读取现有配置。
2. 多个 URL 可正常编辑、保存和再次加载。
3. 非法正则能得到明确提示。
4. 预览结果能展示节点统计和保留列表。
5. 固定订阅地址可一键复制。
6. 重置 token 后地址更新。

### 集成验证

本地启动服务后，需要至少验证以下流程：

1. 使用模拟上游订阅作为输入源。
2. 通过管理页保存配置。
3. 使用固定订阅地址请求公开接口。
4. 返回结果为可用的 Clash YAML，且只包含期望代理组。

## 预期改动范围

预计涉及以下文件或模块：

1. `client/src/App.vue`：新增菜单入口。
2. `client/src/router/index.js`：新增订阅转换页面路由。
3. `client/src/views/Subscription.vue`：实现订阅转换管理页。
4. `client/src/util.js` 或新增前端工具文件：补充复制、表单转换或展示辅助逻辑。
5. `server-node/app/http-router.js`：新增管理接口和公开订阅接口。
6. `server-node/app` 下新增订阅转换相关模块：负责配置读写、订阅拉取、格式识别、节点解析、过滤、YAML 生成、缓存。
7. `server-node/package.json`：按需增加 YAML 解析或生成依赖。
8. `server-node` 下新增测试文件：覆盖转换逻辑。

## 风险与约束

1. 不同订阅格式差异较大，首批实现应聚焦常见来源，避免一次性追求全格式覆盖。
2. 某些 URI 协议字段组合复杂，转换时应优先保证常见节点可用，再逐步补充边缘格式。
3. 上游订阅可能不稳定，因此保存配置与预览必须解耦。
4. 公开订阅接口实时拉取会引入外部依赖波动，因此需要短时缓存与明确错误信息。

## 结论

本次功能以“单一固定订阅地址 + 后台可编辑配置 + 访问时实时转换”为核心模型，新增一个独立前端管理页和一组后端接口。后端将多个上游订阅统一解析为内部节点结构，应用包含/屏蔽正则后，生成一份仅保留“订阅节点”“自动选择”“故障转移”的精简 Clash YAML。该方案兼顾了用户体验、实现复杂度和当前项目结构的可维护性。
